# Phase 24 — Real Native Host Projects for Mobile (D106)

> Status: PLANNED (not started)
> Started: —
> Completed: —
> Decision: **D106** — winit cannot own the iOS app; mobile needs a real,
> editable native host project generated by `tzr new`.

## Why This Phase

Phase 20–22's iOS/Android bring-up proved the rendering engine compiles and
runs on-device (compositor, safe-area, touch input, `tzr run --target ios`).
That was the right first step, but the result — an `Info.plist` next to a raw
executable, no `.xcodeproj` — is not a viable end-state. Verified directly in
winit 0.30.13's source: `EventLoop::run_app` calls `UIApplicationMain` on iOS
and winit generates its own implicit `AppDelegate`. There is no supported way
to run winit inside an `AppDelegate` the user controls. That makes an entire
class of real-app requirements structurally unreachable as long as winit owns
the iOS process: push-notification registration, deep links / universal
links, background tasks, and anything else that lives in AppDelegate/
SceneDelegate methods. Xcode-editable entitlements, capabilities, icons, and
signing are also unreachable without a real `.xcodeproj`.

Android is not as blocked — winit's Android backend runs on `android-activity`,
which explicitly supports subclassing `NativeActivity`/`GameActivity` in
Kotlin — but we still generate no Gradle project, manifest, or Activity today.

## The Model

- **iOS**: a real `.xcodeproj`, generated by `tzr new`, with our OWN thin
  `AppDelegate`/`SceneDelegate` (Swift) — not winit's. The Rust engine
  compiles to a **staticlib** and is linked into the Xcode target. A small C
  FFI boundary (init / resize / input-event / frame-tick) lets the Swift host
  create the `UIView`/`CAMetalLayer` and drive the Rust render loop inside it.
  Winit's iOS backend is retired for the shipped app (desktop/web keep using
  winit — it works well there and isn't blocked the same way).
- **Android**: a real Gradle project — `build.gradle`, `AndroidManifest.xml`,
  a `MainActivity.kt` subclassing `GameActivity`/`NativeActivity`. The Rust
  engine compiles to a **cdylib (.so)**, called via JNI, rendering into a
  `SurfaceView`. Winit can still be used here.
- **`tzr run --target ios|android`** invokes the real toolchains
  (`xcodebuild`, `./gradlew`) against the generated project instead of our
  hand-rolled `codesign`/`simctl` bundling from Phase 20–22 — the generated
  project becomes the source of truth; `tzr` drives it, doesn't replace it.
- This is the "bare" packaging style from D104, done for real; the deferred
  "harness generator" choice in D104 is resolved by this decision.

## Steps

### Step 1 — Design the native-bridge FFI boundary
Define the minimal C ABI the native host (Swift/Kotlin) and the Rust engine
cross: `tzr_engine_init(surface_handle, width, height, scale) -> EngineHandle`,
`tzr_engine_resize(handle, width, height)`, `tzr_engine_input(handle,
InputEventFfi)`, `tzr_engine_frame(handle) -> FrameCommands` (or a direct
render-into-surface call), `tzr_engine_shutdown(handle)`. Keep it small — this
boundary is the seam every future native feature (push token delivery, deep
link URL, permission result) crosses.

Exit: FFI header + Rust `#[no_mangle] extern "C"` surface compiles; a hand-
written throwaway Swift/Kotlin stub can call `init` and get a handle back.

### Step 2 — iOS: thin Swift AppDelegate + Xcode project template
Generate a minimal `.xcodeproj` (AppDelegate.swift, SceneDelegate.swift,
Info.plist, a `UIViewController` owning a `CAMetalLayer`-backed view) that
links the Rust staticlib and drives it through the Step 1 FFI on
`viewDidLoad`/`viewWillLayoutSubviews`/touch callbacks. Reuse the safe-area
math already proven in Phase 20–22 (now driven by real `UIView.safeAreaInsets`
since we own the view hierarchy — no more `outer_size()`/`inner_size()`
workarounds).

Exit: `tzr new app --platforms ios` produces a `.xcodeproj` that opens in
Xcode, builds, and runs the counter example on-device/simulator, with the
same visual correctness already verified in Phase 20–22 (notch/home-indicator
insets, touch input) — via our own AppDelegate instead of winit's.

### Step 3 — Android: Gradle project + MainActivity
Generate `build.gradle`, `AndroidManifest.xml`, `MainActivity.kt` (subclasses
`GameActivity` or `NativeActivity`), loading the Rust cdylib via
`System.loadLibrary` and JNI. Decide whether to keep winit here (subclassing
is supported) or mirror the iOS FFI boundary for consistency — pick based on
how much of Phase 20-22's Android-specific work (once started) can be reused.

Exit: `tzr new app --platforms android` produces a Gradle project that builds
an APK and runs on an emulator/device.

### Step 4 — `tzr run --target ios|android` drives the real toolchains
Replace the Phase 20-22 hand-rolled `codesign`/`simctl install` pipeline with
`xcodebuild -scheme ... -destination ...` (+ `simctl install/launch` only for
the final install/launch step) for iOS, and `./gradlew installDebug` +
`adb shell am start` for Android.

Exit: `tzr run --target ios` / `--target android` work unchanged from the
developer's perspective, but underneath drive a real, user-editable project.

### Step 5 — Permission/capability plumbing proof
Pick one real platform feature unreachable today (e.g. camera permission
prompt, or push-notification registration) and wire it end-to-end: Info.plist
key + Swift AppDelegate code + FFI callback into Rust — proving the native
host model actually delivers what Info.plist-only could not.

Exit: a demo app requests and receives a native permission result in Rust
code, via the new bridge.

## Migration Rule

Desktop and web are UNCHANGED — winit continues to own them; this phase is
mobile-only. The Phase 20-22 hand-rolled iOS harness (`Info.plist` +
`simctl`) is superseded, not deleted immediately — keep it working until
Step 2 lands a working `.xcodeproj` replacement, so there's no dead period
with no working iOS path.

## DO NOT

- Do not attempt to keep winit owning the iOS app lifecycle "for now" while
  building the native host in parallel — the whole point is winit cannot
  reach AppDelegate-level features; there is no partial version of this fix.
- Do not skip the FFI boundary design (Step 1) and start writing Swift/Kotlin
  glue ad hoc — every native feature added later crosses this boundary; get
  its shape right first.
- Do not remove the Phase 20-22 safe-area/touch-input logic — its *math* was
  verified correct on-device; only the surface it's plumbed through changes
  (real `UIView` instead of winit's window).
